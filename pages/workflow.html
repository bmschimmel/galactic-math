How issues enter the system

There are two entry points. The most common is a GitHub issue — anyone with a GitHub account can open one. The Linear–GitHub integration picks it up automatically and places it in Triage within seconds.

The second path is staff creating a Linear issue directly, usually for planned internal work like infrastructure updates or refactors where no external reporter is involved.

Both paths converge at Triage. It doesn't matter which way an issue arrives — the review process is the same.

What happens during triage

The team asks a few questions: Is this reproducible? Is it in scope for a K-6 math trainer? Does it affect the core experience kids actually use?

Issues stay in Triage until someone consciously moves them — there's no timeout. The goal is that nothing gets accidentally dropped.

Triage is a forcing function: every issue needs a human decision before any work starts. That keeps the backlog clean and intentional.

Why we close issues

Not every idea fits. This game is intentionally focused and simple — feature creep makes it harder to maintain and harder for kids to use.

A decline always gets a comment. The reporter put in effort; they deserve to know why. Common reasons: out of scope, duplicate, too complex for the single-file constraint.

Backlog vs. To Do

Backlog = accepted but not yet scheduled. To Do = ready to start right now.

The claude label signals that the issue is well-defined enough for Claude Code to implement autonomously — no ambiguity, no external credentials needed, code-only change.

Claude only picks up issues that are both in To Do state and have the claude label. Both conditions must be true.

Branch naming convention

Format: idt-[issue-number]-short-description

The idt- prefix links the branch to the Linear issue in some integrations and makes the git log instantly scannable — you can see what every branch is doing at a glance.

Never push directly to main. Branch protection is enforced — direct pushes will be rejected.

Always pull main first. If another PR merged while you were working, rebase rather than merge to keep history clean.
Build Solution
details

Commit: IDT-XX description in imperative lowercase · one commit per logical change.
The single-file constraint

The entire game lives in index.html — HTML, CSS, and JS together. This is intentional: no build pipeline, no npm, no bundler. Anyone can open the file in a browser and it just works, even offline.

Hard rules:

· No external dependencies (no npm, no CDN libraries)
· No localStorage or cookies — no data persists between sessions
· All sounds via Web Audio API — no audio files
· CSS variables only for colors — defined in :root
· No <form> tags — use button onClick handlers

The commit ID format IDT-XX is parsed by the Linear integration to auto-link commits to the issue. Without it, Linear won't show your commits in the issue timeline.

Why there's no test runner

This project deliberately has no build step and no test framework. The game is a single HTML file — the "test" is opening it in a browser and clicking through it.

This isn't laziness; it's a constraint that keeps the project approachable. A parent or teacher who wants to verify a fix can do it without installing anything.

The magic of "Fixes IDT-XX"

Including Fixes IDT-XX in the PR description is parsed by the Linear–GitHub integration. When the PR merges to main, Linear automatically moves the issue to Done. Without it, you'd have to close the issue manually.

Opening the PR also automatically moves Linear to In Review — no manual state change needed.

Claude never merges its own PR. It pushes the branch and opens the PR, then stops. A human must approve and merge.

For UI changes, note in the PR body which browser you tested in. Reviewers often skip re-testing if they trust the test notes.

Review + Test
details
Human reviewer opens the branch locally, verifies the checklist, then approves or requests changes.
What a good review looks like

Clone/pull the branch, open index.html in a browser, and play through the affected feature. The checklist above is the minimum bar.

Also check: does the change feel right for a 6-year-old? Is the language simple and encouraging? Does anything feel punishing or frustrating?

Code review: look for new colors that aren't CSS variables, any localStorage calls, or external script tags that would break offline use.

If you request changes, be specific in the PR comment. Claude will re-read it and loop back — vague feedback leads to vague fixes.

What happens on merge

Three things happen automatically the moment the PR merges to main:

1. Linear issue closes — parsed from "Fixes IDT-XX" in the PR body
2. Cloudflare Pages build kicks off — ~60 second deploy
3. Live site updates at galactic-math.pages.dev

No manual deploy step. No staging environment. Merge = ship.

Cloudflare Pages Deploy
details
Auto-deploys on every merge to main. Live within ~60 seconds at galactic-math.pages.dev
How CF Pages works here

Cloudflare Pages watches the main branch. Any push triggers a build and deploy. Because there's no build step (no npm, no bundler), the "build" is essentially just copying files to Cloudflare's edge network.

The result is a globally distributed static site with no server to maintain, no cold starts, and free SSL.

The Cloudflare Worker (feedback handler) is a separate deployment — it lives in worker/ and is deployed independently via Wrangler, not automatically on git push.

details

What's actually served

The game runs entirely in the browser — no server-side logic for gameplay. All math question generation, scoring, sounds, and animations happen client-side in index.html.

The only server-side piece is the Cloudflare Worker that handles feedback form submissions. Everything else is static files on CF Pages.

Because there's no server, the game works offline after first load. Google Fonts is the only external dependency — it's UI polish, not functionality.

Who this is built for

Kids ages 5–12 are the primary users — they play the game directly. Every UI decision should consider a 6-year-old using a tablet.

Parents and teachers are secondary users — they share it, assign it, and sometimes file bugs. The feedback form is intentionally in the footer so kids don't see it as part of the game.

Keep UI language simple, fun, and encouraging. Avoid anything that feels punishing. "Youngling 🌱" for a low score is still a positive frame — it's a Star Wars rank, not a failing grade.

Cloudflare Worker — galactic-math-feedback
details
worker/feedback-worker.js — receives form POST, calls GitHub API to create a pre-filled issue.
Observability & invocation logs enabled via wrangler.toml.
Why a Worker instead of a form action

The feedback form can't POST directly to GitHub because the GitHub API token would be exposed in client-side code. The Cloudflare Worker runs server-side, holds the token securely, and acts as a proxy.

Observability logs (enabled in wrangler.toml) record every invocation so we can debug failures — e.g. if the GitHub API token expires or rate limits are hit.

This Worker is deployed separately from CF Pages using wrangler deploy — it doesn't auto-deploy on git push to main.

The Worker is a separate Cloudflare service from the Pages site. Changes to worker/feedback-worker.js require a manual wrangler deploy from the worker/ directory.

details

Closing the loop

The Worker creates a GitHub issue with the feedback text pre-filled in a structured format. The Linear–GitHub integration immediately picks it up and places it in Triage — exactly the same as if a user had opened the issue manually on GitHub.

This means user feedback from the live site feeds directly into the development workflow without any manual forwarding or copy-pasting.

The loop is complete: users play → submit feedback → issue in Triage → reviewed → built by Claude → deployed → users play the fix.
